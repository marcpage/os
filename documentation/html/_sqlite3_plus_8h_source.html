00325 
00326 After a prepared statement has been prepared using either sqlite3_prepare_v2() or
00327         sqlite3_prepare16_v2() or one of the legacy interfaces sqlite3_prepare() or sqlite3_prepare16(),
00328         this function must be called one or more times to evaluate the statement.
00329 The details of the behavior of the sqlite3_step() interface depend on whether the statement was
00330         prepared using the newer "v2" interface sqlite3_prepare_v2() and sqlite3_prepare16_v2() or the
00331         older legacy interface sqlite3_prepare() and sqlite3_prepare16(). The use of the new "v2"
00332         interface is recommended for new applications but the legacy interface will continue to be
00333         supported.
00334 In the legacy interface, the return value will be either SQLITE_BUSY, SQLITE_DONE, SQLITE_ROW,
00335         SQLITE_ERROR, or SQLITE_MISUSE. With the "v2" interface, any of the other result codes or
00336         extended result codes might be returned as well.
00337 SQLITE_BUSY means that the database engine was unable to acquire the database locks it needs to do
00338         its job. If the statement is a COMMIT or occurs outside of an explicit transaction, then you can
00339         retry the statement. If the statement is not a COMMIT and occurs within a explicit transaction
00340         then you should rollback the transaction before continuing.
00341 SQLITE_DONE means that the statement has finished executing successfully. sqlite3_step() should not
00342         be called again on this virtual machine without first calling sqlite3_reset() to reset the
00343         virtual machine back to its initial state.
00344 If the SQL statement being executed returns any data, then SQLITE_ROW is returned each time a new
00345         row of data is ready for processing by the caller. The values may be accessed using the column
00346         access functions. sqlite3_step() is called again to retrieve the next row of data.
00347 SQLITE_ERROR means that a run-time error (such as a constraint violation) has occurred.
00348         sqlite3_step() should not be called again on the VM. More information may be found by calling
00349         sqlite3_errmsg(). With the legacy interface, a more specific error code (for example,
00350         SQLITE_INTERRUPT, SQLITE_SCHEMA, SQLITE_CORRUPT, and so forth) can be obtained by calling
00351         sqlite3_reset() on the prepared statement. In the "v2" interface, the more specific error code
00352         is returned directly by sqlite3_step().
00353 SQLITE_MISUSE means that the this routine was called inappropriately. Perhaps it was called on a
00354         prepared statement that has already been finalized or on one that had previously returned
00355         SQLITE_ERROR or SQLITE_DONE. Or it could be the case that the same database connection is being
00356         used by two or more threads at the same moment in time.
00357 Goofy Interface Alert: In the legacy interface, the sqlite3_step() API always returns a generic
00358         error code, SQLITE_ERROR, following any error other than SQLITE_BUSY and SQLITE_MISUSE. You must
00359         call sqlite3_reset() or sqlite3_finalize() in order to find one of the specific error codes that
00360         better describes the error. We admit that this is a goofy design. The problem has been fixed
00361         with the "v2" interface. If you prepare all of your SQL statements using either
00362         sqlite3_prepare_v2() or sqlite3_prepare16_v2() instead of the legacy sqlite3_prepare() and
00363         sqlite3_prepare16() interfaces, then the more specific error codes are returned directly by
00364         sqlite3_step(). The use of the "v2" interface is recommended.

00370 The sqlite3_reset() function is called to reset a prepared statement object back to its initial
00371         state, ready to be re-executed. Any SQL statement variables that had values bound to them using
00372         the sqlite3_bind_*() API retain their values. Use sqlite3_clear_bindings() to reset the bindings.
00373 The sqlite3_reset(S) interface resets the prepared statement S back to the beginning of its program.
00374 If the most recent call to sqlite3_step(S) for the prepared statement S returned SQLITE_ROW or
00375         SQLITE_DONE, or if sqlite3_step(S) has never before been called on S, then sqlite3_reset(S)
00376         returns SQLITE_OK.

00378         sqlite3_reset(S) returns an appropriate error code.

00380         S.
